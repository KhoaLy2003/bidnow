# CI/CD Enhancement Design

**Date:** 2026-06-28  
**Status:** Approved

## Overview

Enhance the BidNow CI/CD pipeline from a minimal build-only setup to a full quality, security, and deployment pipeline. Five separate workflow files cover distinct concerns; Docker and deploy workflows are fully defined but disabled until a target environment is ready.

## Goals

- Run the full backend test suite (unit + BDD/Cucumber) on every PR
- Add frontend linting as a fast pre-build gate
- Introduce security scanning using free, GitHub-native tools
- Define Docker build and CD deploy workflows so enabling them later is a one-line change
- Deliver CI failure notifications via email (GitHub's built-in mechanism)

## Non-Goals

- Frontend test execution (deferred — no tests exist yet)
- Docker image publishing and deployment (defined but disabled)
- Paid scanning tools (Snyk, SonarCloud, etc.)

## Architecture

### Workflow File Breakdown

| File | Trigger | Status |
|---|---|---|
| `backend-ci.yml` | PR/push → `main`, `develop` on `backend/**` | Active |
| `frontend-ci.yml` | PR/push → `main`, `develop` on `frontend/**` | Active |
| `security.yml` | PR/push → `main`, `develop` + weekly cron | Active |
| `docker-build.yml` | PR/push → `main`, `develop` | Disabled (`if: false`) |
| `cd-deploy.yml` | Push → `main` only | Disabled (`if: false`) |

The existing `claude-review.yml` is unchanged.

---

### Backend CI (`backend-ci.yml`)

Replaces the current build-only workflow with a two-stage pipeline.

**Jobs:**

1. **`build`**
   - `mvn package -DskipTests` in the `backend/` directory
   - Java 17 Temurin, Maven cache enabled

2. **`test`** (blocked by `build`)
   - `mvn verify` — runs unit tests and BDD/Cucumber tests for all services
   - Test results published to PR via `dorny/test-reporter` (JUnit XML format)
   - Failure blocks PR merge

3. **`coverage-report`** (blocked by `test`)
   - JaCoCo XML/HTML report generated by Maven
   - Uploaded as a GitHub Actions artifact for manual review

**Notifications:** GitHub sends email to the committer automatically on workflow failure. No additional configuration required.

---

### Frontend CI (`frontend-ci.yml`)

Adds a lint gate before the existing build step.

**Jobs:**

1. **`lint`**
   - `npm run lint` in the `frontend/` directory
   - Fails fast; build is skipped if lint fails

2. **`build`** (blocked by `lint`)
   - `npm run build` — same as current workflow
   - Node 20, npm cache enabled

A `test` job can be added later blocked by `build` once frontend tests exist.

---

### Security (`security.yml`)

Three parallel jobs using free, GitHub-native tooling.

**Jobs:**

1. **`codeql`**
   - GitHub CodeQL action scanning Java and JavaScript/TypeScript
   - Detects code-level vulnerabilities (SQL injection, XSS, path traversal, etc.)
   - Results surface in the GitHub Security tab

2. **`owasp-dependency-check`**
   - OWASP Dependency-Check Maven plugin scans all `pom.xml` dependencies for CVEs
   - HTML report uploaded as a GitHub Actions artifact
   - Configured with a CVSS failure threshold (e.g., score ≥ 7 fails the job)

3. **`dependency-review`**
   - GitHub `actions/dependency-review-action`
   - Runs only on `pull_request` events
   - Blocks PRs that introduce dependencies with known high-severity vulnerabilities

**Dependabot (`.github/dependabot.yml`):**
- Weekly automated PRs for outdated Maven and npm dependencies
- Separate PR per ecosystem so updates are reviewable independently

**Schedule:** Weekly cron (`0 0 * * 1` — every Monday at midnight UTC) catches CVEs disclosed between code changes.

---

### Docker Build (`docker-build.yml`) — Disabled

Defined but all jobs gated with `if: false`.

**When enabled:**
- Matrix strategy builds Docker images for all 8 services in parallel: `api-gateway`, `auction-service`, `bidding-service`, `discovery-service`, `identity-service`, `media-service`, `user-service`, `wallet-service`
- Images pushed to GitHub Container Registry (GHCR) — free, no external account needed
- Tagged with commit SHA (`ghcr.io/org/service:sha`) for traceability

**To enable:** Remove `if: false` from the matrix job and ensure `GITHUB_TOKEN` has `packages: write` permission (already available by default in GitHub Actions).

---

### CD Deploy (`cd-deploy.yml`) — Disabled

Defined but all jobs gated with `if: false`. Triggers only on push to `main`.

**When enabled:**
- Depends on `docker-build` workflow completing successfully (via `workflow_run` trigger)
- Placeholder steps: SSH to target server → `docker pull` → `docker compose up -d`
- Requires secrets: `DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_SSH_KEY`

**To enable:** Remove `if: false`, fill in deploy secrets in GitHub repository settings, and update the compose file path to match the server layout.

---

## Error Handling

- Backend build failure → `test` and `coverage-report` jobs are skipped automatically (job dependency)
- Lint failure → `build` job skipped (saves CI minutes)
- Security job failures do not block backend/frontend CI (separate workflow, separate status check)
- OWASP false positives can be suppressed via a `suppression.xml` file committed to `backend/`

## Testing Strategy

The CI pipeline itself is validated by:
- Intentionally breaking a test locally and verifying the `test` job fails on PR
- Intentionally introducing a lint error and verifying the `lint` job fails
- Reviewing the OWASP HTML artifact after the first successful security run to tune the CVSS threshold

## Enabling Deferred Features

| Feature | Action Required |
|---|---|
| Docker builds | Remove `if: false` in `docker-build.yml` |
| CD deployment | Remove `if: false` in `cd-deploy.yml`, add deploy secrets |
| Frontend tests | Add `test` job to `frontend-ci.yml` blocked by `build` |
| Coverage badge | Add Codecov or similar action to `coverage-report` job |
